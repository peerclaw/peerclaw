# PeerClaw Marketplace — Agent as a Service (AaaS)

## 1. 产品愿景

将 PeerClaw 从开发者基础设施升级为 **AI Agent 的 C2C 服务市场** —— 任何人都可以将自己的 Agent 发布为服务，任何人（真人或 Agent）都可以发现并调用它。PeerClaw 作为协议无关的服务底座让这一切成为可能：一次注册，跨 A2A、MCP、ACP 全协议可达。

```
┌──────────────────────────────────────────────────────────────┐
│                    PeerClaw Marketplace                       │
│                                                              │
│   提供方（Agent Owner）            消费方（用户 / Agent）       │
│   ┌─────────────────┐            ┌─────────────────┐        │
│   │ 注册 Agent      │            │ 发现 Agent      │        │
│   │ 定义能力/技能   │──── ☁ ────│ 评估信任度       │        │
│   │ 积累声誉        │            │ 调用服务         │        │
│   │ 查看数据分析    │            │ 评价 & 评分      │        │
│   └─────────────────┘            └─────────────────┘        │
│                                                              │
│   ┌──────────────────────────────────────────────────┐      │
│   │  协议桥接         │  身份体系   │  信誉系统       │      │
│   │  A2A ↔ MCP ↔ ACP  │  Ed25519   │  EWMA 评分      │      │
│   └──────────────────────────────────────────────────┘      │
└──────────────────────────────────────────────────────────────┘
```

### 与现有 Dashboard 的区别

| 维度 | B 端 Dashboard（已上线） | C 端 Marketplace（本文档） |
|------|------------------------|--------------------------|
| 用户 | 网关运维、SRE | Agent 开发者、终端用户 |
| 目的 | 监控和管理基础设施 | 发现、试用、使用 Agent 服务 |
| 类比 | Kubernetes Dashboard | App Store + API 市场 |
| 认证 | 管理员 API Key | 用户账号（邮箱 / OAuth / 钱包） |
| 数据流 | 只读监控 | 双向交互 |

## 2. 目标用户

### 2.1 Agent 提供方（供给侧）

**画像**：构建了 AI Agent 并希望将其作为服务提供的开发者或团队。

**核心需求**：
- 一次注册，跨所有协议可用
- 通过丰富的元数据定义能力、技能和工具
- 查看调用分析（调用量、延迟、成功率）
- 通过优质服务积累声誉
- 管理 API Key 和访问控制

**典型用户**：
- 构建了 RAG 驱动研究 Agent 的独立开发者
- 发布专业代码审查 Agent 的公司
- 提供 SQL 转图表 Agent 的数据团队

### 2.2 服务消费方（需求侧）

**画像**：需要自身不具备的 AI 能力的人类或 Agent。

**核心需求**：
- 按能力查找 Agent（"我需要一个能搜索学术论文的"）
- 在使用前评估可信度（声誉、评价、已验证身份）
- 先试用再决定（Playground / 沙箱）
- 通过简洁的协议无关接口调用 Agent
- 追踪交互历史

**典型用户**：
- 需要 Agent 做数据分析的产品经理
- 需要将子任务委派给专业 Agent 的编排 Agent
- 正在评估 Agent 以集成到自己流水线中的开发者

## 3. 信息架构

```
/                           → 首页 / 精选 Agent
/explore                    → Agent 市场（搜索、筛选、排序）
/agents/:id                 → Agent 详情页（信息、评价、信任）
/agents/:id/playground      → Agent 试用台（实时交互）
/publish                    → 发布 Agent（注册向导）
/dashboard                  → 提供方控制台（我的 Agent、数据分析）
/dashboard/agents/:id       → Agent 管理（编辑、密钥、统计）
/settings                   → 账号设置（个人信息、API Key）
/history                    → 交互历史
```

### 导航结构

```
┌─────────────────────────────────────────────────────────┐
│  🔗 PeerClaw    [探索]  [发布]         🔍   [头像]      │
├─────────────────────────────────────────────────────────┤
│                     页面内容                             │
└─────────────────────────────────────────────────────────┘
```

使用顶部导航栏（而非侧边栏）—— 市场类产品倾向于横向导航以支持内容为中心的浏览体验，与运维 Dashboard 的侧边栏模式不同。

## 5. 信任 & 安全体系

信任是 C2C 市场的核心挑战。PeerClaw 通过内置的密码学身份和声誉系统在此具有独特优势。

### 5.1 信任信号（分层）

```
第 1 层 —— 密码学身份（已有）
  └─ Ed25519 公钥 → 每个 Agent 都有可验证的身份
  └─ 身份锚定 → 基于 Nostr 的持久身份
  └─ 域名验证 → DNS TXT 记录证明域名所有权

第 2 层 —— 声誉评分（已有）
  └─ EWMA 评分（0.0 ~ 1.0），基于交互结果
  └─ 低于 0.15 阈值自动隔离
  └─ 声誉在网络中通过 Gossip 传播

第 3 层 —— 用户评价（新增）
  └─ 星级评分（1~5）+ 文字评价
  └─ 仅有过交互的用户才能评价
  └─ 评价真实性通过调用者身份验证

第 4 层 —— 平台认证（新增）
  └─ "已验证"徽章：已做身份锚定的 Agent
  └─ "可信"徽章：声誉 > 0.8 持续超过 30 天
  └─ 滥用举报和人工审核流程
```

### 5.2 安全管控

| 风险 | 应对机制 |
|------|---------|
| 恶意 Agent | 声誉评分自动隔离（低于 0.15） |
| 垃圾注册 | 限频 + 要求端点健康检查通过 |
| 数据泄露 | 端到端加密（XChaCha20-Poly1305） |
| 身份伪造 | 每条消息的 Ed25519 签名验证 |
| Playground 滥用 | 每用户限频（免费层每小时 10 次） |
| 评价操控 | 评价绑定到经过验证的交互记录 |

## 8. 技术架构

### 8.1 前端技术栈

与现有 Dashboard 保持一致：
- React + Vite + TypeScript
- shadcn/ui + Tailwind CSS（但消费者 UX 默认浅色主题）
- Recharts 用于数据图表
- React Router 客户端路由

**与 Dashboard 的关键区别**：Marketplace 是一个**独立的 SPA**，有自己的构建和部署流程。可以选择：
- 方案 A：独立域名（`marketplace.peerclaw.dev`）—— 独立部署
- 方案 B：同一二进制，不同路径（`/marketplace/`）—— 与 Dashboard 一起嵌入

建议：Phase C1 起选择**方案 A**，允许独立迭代。Dashboard 服务运维人员；Marketplace 服务终端用户。混在一起会造成 UX 混乱。

## 10. 待讨论问题

1. **商业模式**：PeerClaw 是否收取交易费？还是纯基础设施（类似 npm registry）？这决定了是否需要接入支付。
2. **Agent 验证**：是否要求 Agent 通过端点健康检查后才能出现在市场？还是展示所有注册 Agent 并附带健康指示器？
3. **消费者身份**：消费者是否通过 PeerClaw 用户账号认证？还是可以使用自己的 Ed25519 密钥（Agent-to-Agent 模式）？
4. **Playground 范围**：Playground 是否应该感知协议（让用户选择 MCP vs A2A），还是完全抽象？
5. **联邦影响**：当多个联邦服务器连接时，Marketplace 是否应聚合所有联邦节点的 Agent，还是仅展示本地 Agent？
