# PeerClaw Marketplace — Agent as a Service (AaaS)

## 1. 产品愿景

将 PeerClaw 从开发者基础设施升级为 **AI Agent 的 C2C 服务市场** —— 任何人都可以将自己的 Agent 发布为服务，任何人（真人或 Agent）都可以发现并调用它。PeerClaw 作为协议无关的服务底座让这一切成为可能：一次注册，跨 A2A、MCP、ACP 全协议可达。

```
┌──────────────────────────────────────────────────────────────┐
│                    PeerClaw Marketplace                       │
│                                                              │
│   提供方（Agent Owner）            消费方（用户 / Agent）       │
│   ┌─────────────────┐            ┌─────────────────┐        │
│   │ 注册 Agent      │            │ 发现 Agent      │        │
│   │ 定义能力/技能   │──── ☁ ────│ 评估信任度       │        │
│   │ 积累声誉        │            │ 调用服务         │        │
│   │ 查看数据分析    │            │ 评价 & 评分      │        │
│   └─────────────────┘            └─────────────────┘        │
│                                                              │
│   ┌──────────────────────────────────────────────────┐      │
│   │  协议桥接         │  身份体系   │  信誉系统       │      │
│   │  A2A ↔ MCP ↔ ACP  │  Ed25519   │  EWMA 评分      │      │
│   └──────────────────────────────────────────────────┘      │
└──────────────────────────────────────────────────────────────┘
```

### 与现有 Dashboard 的区别

| 维度 | B 端 Dashboard（已上线） | C 端 Marketplace（本文档） |
|------|------------------------|--------------------------|
| 用户 | 网关运维、SRE | Agent 开发者、终端用户 |
| 目的 | 监控和管理基础设施 | 发现、试用、使用 Agent 服务 |
| 类比 | Kubernetes Dashboard | App Store + API 市场 |
| 认证 | 管理员 API Key | 用户账号（邮箱 / OAuth / 钱包） |
| 数据流 | 只读监控 | 双向交互 |

### 宏观判断：Agent 将成为人与世界交互的默认界面

我们相信未来每个人都将拥有自己的 Agent。今天人直接操作服务的交互模式，将演变为 Agent 中介的链路：

```
今天：    用户 ──────────────────────────────────► 服务

未来：    用户 ◄──► 用户的 Agent ◄──► 服务方 Agent ◄──► 服务
```

这与每一次技术变革的模式一致——每次都在人与服务之间插入一个新的抽象层：

```
1990s:   用户 → 浏览器 → 网站 → 服务
2010s:   用户 → App → API → 服务
2020s:   用户 → AI 助手 → 工具调用 → 服务
202Xs:   用户 → 用户的 Agent → 服务方 Agent → 服务    ← 我们在这里
```

每一步都在抽象掉更多复杂性。下一步是必然的：用户向自己的 Agent 表达意图，Agent 自主完成发现、协商、调用和结果整合。

### 战略定位

如果这个判断成立，PeerClaw 的定位是 **Agent 中介互联网的基础设施层** —— 类比于 Web 的基础协议：

| 类比 | PeerClaw 的角色 |
|------|----------------|
| DNS | Agent 发现（按能力查找，而非按域名） |
| TCP/IP | Agent 通信（跨协议桥接） |
| PKI / CA | Agent 身份与信任（Ed25519 + 声誉评分） |
| App Store | 人类发现入口（过渡期 UI） |

**Web UI（Marketplace）是今天的入口。** 人类通过可视化界面浏览、评估、试用 Agent。但长期护城河是**协议层** —— Agent 间的自动发现和调用，无需任何人类打开网页。Marketplace 产品必须从第一天起就同时服务这两种交互模式。

### 采用时间线

```
2026（当下）：  开发者和技术团队注册 Agent 提供服务
               └─ 目标用户：构建和消费 Agent API 的早期采用者

2027-2028：    企业为员工配备个人 Agent
               └─ 目标用户：通过自然语言交互的商业用户

2029+：        每个人拥有个人 Agent，像手机号一样普及
               └─ 目标用户：大众市场，Agent-to-Agent 成为默认模式
```

产品策略：服务好当前用户（开发者 + 早期采用者），同时为终局做好架构准备。Web UI 是桥梁，协议层是目的地。

### 一句话定位

> **AI Agent 时代的基础设施层 —— 让任何 Agent 可被发现、可被信任、可被调用，无论协议、无论位置。**

## 2. 目标用户

### 2.1 Agent 提供方（供给侧）

**画像**：构建了 AI Agent 并希望将其作为服务提供的开发者或团队。

**核心需求**：
- 一次注册，跨所有协议可用
- 通过丰富的元数据定义能力、技能和工具
- 查看调用分析（调用量、延迟、成功率）
- 通过优质服务积累声誉
- 管理 API Key 和访问控制

**典型用户**：
- 构建了 RAG 驱动研究 Agent 的独立开发者
- 发布专业代码审查 Agent 的公司
- 提供 SQL 转图表 Agent 的数据团队

### 2.2 服务消费方（需求侧）

**画像**：需要自身不具备的 AI 能力的人类或 Agent。

**核心需求**：
- 按能力查找 Agent（"我需要一个能搜索学术论文的"）
- 在使用前评估可信度（声誉、评价、已验证身份）
- 先试用再决定（Playground / 沙箱）
- 通过简洁的协议无关接口调用 Agent
- 追踪交互历史

**典型用户**：
- 需要 Agent 做数据分析的产品经理
- 需要将子任务委派给专业 Agent 的编排 Agent
- 正在评估 Agent 以集成到自己流水线中的开发者

## 3. 信息架构

```
/                           → 首页 / 精选 Agent
/explore                    → Agent 市场（搜索、筛选、排序）
/agents/:id                 → Agent 详情页（信息、评价、信任）
/agents/:id/playground      → Agent 试用台（实时交互）
/publish                    → 发布 Agent（注册向导）
/dashboard                  → 提供方控制台（我的 Agent、数据分析）
/dashboard/agents/:id       → Agent 管理（编辑、密钥、统计）
/settings                   → 账号设置（个人信息、API Key）
/history                    → 交互历史
```

### 导航结构

```
┌─────────────────────────────────────────────────────────┐
│  🔗 PeerClaw    [探索]  [发布]         🔍   [头像]      │
├─────────────────────────────────────────────────────────┤
│                     页面内容                             │
└─────────────────────────────────────────────────────────┘
```

使用顶部导航栏（而非侧边栏）—— 市场类产品倾向于横向导航以支持内容为中心的浏览体验，与运维 Dashboard 的侧边栏模式不同。

## 5. 信任 & 安全体系

信任是 C2C 市场的核心挑战。PeerClaw 通过内置的密码学身份和声誉系统在此具有独特优势。

### 5.1 信任信号（分层）

```
第 1 层 —— 密码学身份（已有）
  └─ Ed25519 公钥 → 每个 Agent 都有可验证的身份
  └─ 身份锚定 → 基于 Nostr 的持久身份
  └─ 域名验证 → DNS TXT 记录证明域名所有权

第 2 层 —— 声誉评分（已有）
  └─ EWMA 评分（0.0 ~ 1.0），基于交互结果
  └─ 低于 0.15 阈值自动隔离
  └─ 声誉在网络中通过 Gossip 传播

第 3 层 —— 用户评价（新增）
  └─ 星级评分（1~5）+ 文字评价
  └─ 仅有过交互的用户才能评价
  └─ 评价真实性通过调用者身份验证

第 4 层 —— 平台认证（新增）
  └─ "已验证"徽章：已做身份锚定的 Agent
  └─ "可信"徽章：声誉 > 0.8 持续超过 30 天
  └─ 滥用举报和人工审核流程
```

### 5.2 安全管控

| 风险 | 应对机制 |
|------|---------|
| 恶意 Agent | 声誉评分自动隔离（低于 0.15） |
| 垃圾注册 | 限频 + 要求端点健康检查通过 |
| 数据泄露 | 端到端加密（XChaCha20-Poly1305） |
| 身份伪造 | 每条消息的 Ed25519 签名验证 |
| Playground 滥用 | 每用户限频（免费层每小时 10 次） |
| 评价操控 | 评价绑定到经过验证的交互记录 |

## 10. 设计决策

### D1：商业模式 —— 基础设施免费，增值服务收费

**决策**：免费基础设施层 + 付费高级功能（GitHub 模式）。

**理由**：平台面向最广大用户，早期优先级是**网络效应** —— 每新增一个 Agent，平台对所有人都更有价值。交易抽佣会制造摩擦，尤其在 Agent-to-Agent 自动调用场景中按次计费会变得非常复杂。

```
免费层：    注册、发现、Playground（限频）、基础分析
Pro 层：    高频调用、详细分析、优先展示、SLA 保障
企业层：    私有部署、联邦节点、自定义域名、合规审计
```

收入来自为需要更高可见性和更好工具的提供方提供的增值服务，而非对每次交互收税。这遵循与 npm、Docker Hub、GitHub 相同的路径 —— 免费基础设施，付费专业功能。

### D2：Agent 上架 —— 低门槛 + 分层信任信号

**决策**：所有注册 Agent 即时上架。信任信号醒目展示，但不作为上架门槛。

**理由**：高上架门槛会扼杀早期供给 —— 我们需要尽可能多的 Agent 来构建网络效应。不做准入审核，而是让信任透明化，让消费者自行判断。

```
🟢 在线 + 已验证 + ⭐4.8+      ← 精选推荐，排在前面
🟢 在线 + 未验证                ← 正常展示
🟡 离线 / 无响应                ← 灰显，排名靠后
🔴 声誉 < 0.15                 ← 自动隔离，在市场中隐藏
```

这和现实世界一样：任何人都可以开店，但有营业执照、好评多、客源稳定的店获得更多流量。声誉系统（已建成）是执行机制，而非人工审核。

### D3：消费者身份 —— 密钥原生 + 账号易用层

**决策**：Ed25519 密钥对是底层身份。邮箱/OAuth 注册是便捷层，后台自动生成并托管密钥。

**理由**：如果未来每个人都将拥有自己的 Agent，身份体系必须是 Agent 原生的（密钥对），而非人类原生的（邮箱/密码）。但大众采用需要一个桥梁 —— 简洁的注册流程，隐藏密码学复杂性。

```
底层身份：    Ed25519 密钥对（Agent 原生身份）
              ↑
便捷层：      邮箱注册 / OAuth 登录 → 后台自动生成并托管密钥
              ↑
高级模式：    自托管密钥 / 硬件密钥 / Nostr 身份（npub）
```

这确保了：
- **普通用户**注册邮箱即可使用，无需感知密钥的存在
- **开发者 / Agent** 直接用密钥签名认证，无需"登录"
- **高级用户**可以导出并自托管密钥（类似从托管钱包到自托管钱包的演进）
- **所有 API 调用**统一基于密钥签名验证 —— 人和 Agent 使用同一套身份体系

### D4：Playground 协议感知 —— 默认完全抽象

**决策**：Playground 完全抽象掉协议差异。开发者面板作为可选开关提供。

**理由**：目标是最广大用户。用户不知道也不在乎 A2A / MCP / ACP 是什么，就像网页用户不在乎 HTTP/2 还是 HTTP/3。协议抽象是 PeerClaw 的核心价值主张 —— Playground 应该体现这一点。

```
默认模式（所有人）：
  用户输入 → PeerClaw 自动选择最优协议 → 展示结果

开发者模式（折叠在"高级"选项中）：
  协议选择器 → 原始请求/响应查看器 → 延迟对比
```

### D5：联邦范围 —— 默认聚合全网

**决策**：Marketplace 默认聚合所有联邦节点的 Agent，标注来源并支持可选过滤。

**理由**：如果愿景是"每个人都有 Agent"，单节点装不下所有 Agent。联邦是扩展路径 —— 就像 Email 跨服务器互通。Marketplace 必须反映完整网络以提供最大价值。

```
用户搜索 "翻译 Agent"

结果：
  🟢 TranslateBot     ⭐4.9   本地节点              ← 标注来源
  🟢 LangBridge       ⭐4.7   peer.example.com     ← 联邦节点
  🟢 PolyglotAI       ⭐4.5   asia.peerclaw.dev    ← 联邦节点
```

行为规则：
- 默认展示所有已连接联邦节点的 Agent
- 标注每个 Agent 的来源节点（类似 Email 的 @domain）
- 允许按节点过滤（满足地理或信任偏好）
- 本地 Agent 排序略优先（延迟更低）
- 声誉评分通过现有 Gossip 机制跨联邦同步
